the Lean 4 proof that the libcrux-iot ML-DSA NTT implementation
(`libcrux-iot/ml-dsa/src/`) computes the same functions as the clean `Z_q`

## What builds today

`LibcruxIotMlDsa.lean` imports only `Spec/{Parameters,Montgomery,Pure,Lift}` —
the self-contained clean-`Z_q` reference. `Spec/Parameters` and
`Spec/Montgomery` pin every domain constant with elaboration-time `#guard`
(numerically verified). `Spec/Pure` is a careful hand-translation of the hacspec
NTT that **Phase 1 must `lake build` + `plausible`-validate** before use.

## How ML-DSA differs from the ML-KEM reference

q = 8 380 417 (vs 3329); `i32` coefficients + Montgomery `R = 2³²` (vs `i16`/2¹⁶);
8-lane SIMD units, 32 per polynomial; a **complete 8-layer** NTT (vs 7-layer
incomplete + base multiply) split **within-unit (layers 0–2)** vs
**across-unit (layers 3–7)**; plain pointwise multiply; the spec is clean `Z_q`
(no Montgomery), so the equivalence needs **R-factor reconciliation** (lift ×R⁻¹,
iNTT finalize ×41978 = R²·256⁻¹). The impl carries **no hax annotations**, so
coefficient-bound invariants are supplied in-Lean (re-derived empirically — the
upstream-libcrux F* bounds are structural hints, not values to copy).
